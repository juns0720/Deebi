# SPEC.md — Deebi 제품 스펙

이 문서는 Deebi가 **무엇을 만드는 제품인지** 정의하는 단일 기준이다. 에이전트는 기능을 구현할 때 이 문서를 최우선으로 참조한다. 기술 구현 방식은 `docs/ARCHITECTURE.md`, 데이터 구조는 `docs/DATA_MODEL.md`, API 계약은 `docs/API_CONTRACTS.md`, 화면과 UX는 `docs/UI_UX.md`를 따른다.

---

## 1. 제품 정체성

| 항목 | 내용 |
|---|---|
| 서비스명 | Deebi |
| 제품 형태 | GitHub 활동 기반 웹 대시보드 |
| 대상 플랫폼 | 데스크톱 웹 + 모바일 웹 |
| 초기 언어 | 한국어 단일 |
| 핵심 입력 | 로그인 사용자의 GitHub 커밋 활동 |
| 핵심 출력 | 건강도, 포인트, 도트 캐릭터 상태, 아이템, 룸 멤버 비교, 공동 룸 채팅 |

## 2. 한 줄 요약

개발자가 GitHub에 커밋을 꾸준히 하면 도트 캐릭터가 건강해지고, 방치하면 상태가 나빠지는 웹 서비스. 커밋 활동으로 포인트를 모아 뽑기로 도트 아이템을 얻고, 룸에서 다른 사용자와 캐릭터 상태를 함께 보며 간단히 대화한다.

## 3. 핵심 컨셉

- **방치 = 악화, 활동 = 회복.** 이 문장이 제품의 중심이다.
- GitHub 활동은 숫자 대시보드에 그치지 않고, 캐릭터의 표정과 상태로 즉시 변환된다.
- 포인트와 뽑기는 커밋 습관을 유지하게 하는 가벼운 보상이다.
- 룸은 친구 요청/수락 없이도 서로 비교하고 자랑하고 대화할 수 있는 최소 소셜 구조다.
- 사용자가 매일 들어오고 싶게 만들되, 복잡한 게임 시스템으로 번지지 않는다.

## 4. 사용자와 사용 상황

### 4.1 핵심 사용자

- 개인 프로젝트를 꾸준히 하고 싶은 개발자
- GitHub 잔디를 관리하지만 단순 그래프보다 재미있는 피드백을 원하는 개발자
- 스터디, 팀, 동아리에서 서로의 개발 루틴을 가볍게 확인하고 싶은 사람

### 4.2 주요 사용 상황

- 오늘 커밋을 했는지 확인한다.
- 캐릭터 상태가 나빠졌는지 보고 커밋 동기를 얻는다.
- 포인트가 모였을 때 뽑기로 아이템을 얻는다.
- 룸 링크를 공유해 팀원들의 캐릭터 상태를 비교하고 짧게 대화한다.
- 모바일에서 빠르게 내 상태와 룸 상태를 확인한다.

## 5. 제품 루프

1. 사용자가 GitHub로 로그인한다.
2. 서버가 GitHub 활동을 동기화한다.
3. 활동량에 따라 건강도와 포인트가 갱신된다.
4. 건강도는 도트 캐릭터의 상태로 표현된다.
5. 포인트가 100 이상이면 뽑기를 할 수 있다.
6. 뽑은 아이템을 슬롯에 장착해 캐릭터를 꾸민다.
7. 룸에 참여해 다른 사용자와 캐릭터 상태를 비교한다.
8. 공동 룸 안에서 친구들과 간단한 텍스트 채팅을 한다.
9. 상태 악화, 룸 비교, 친구 대화가 다음 커밋 동기로 이어진다.

## 7. 의도적으로 제외한 것 (스코프 보호)

에이전트는 아래 기능을 구현하지 않는다. 필요해 보여도 코드로 만들지 말고 제안만 한다.

- 친구 요청·수락·알림 시스템
- 팔로우, 차단, DM 같은 소셜 그래프 기능
- 신고/차단/멘션/파일 첨부/읽음 표시 같은 고급 채팅 기능
- 아이템 상점
- 직접 구매
- 아이템 거래
- 아이템 강화·합성
- 캐릭터 레벨, 경험치, 전투력, 스탯 시스템
- 룸 밖에서 동작하는 전역 채팅 또는 개인 DM
- 결제, 인앱 구매
- 푸시 알림
- 이메일 알림
- 다국어
- 런타임 AI 이미지 생성
- 실제 광고 SDK 또는 광고 네트워크 연동
- cron 기반 상시 백그라운드 동기화
- 광고 클릭 유도 또는 광고 노출을 인위적으로 늘리기 위한 자동화

## 8. MVP 기능

### 8.1 랜딩

- 서비스 이름 Deebi를 첫 화면에서 명확히 보여준다.
- GitHub 로그인 진입점을 제공한다.
- 제품의 핵심 루프를 짧게 보여준다.
- 로그인 전에도 제품 컨셉을 이해할 수 있어야 한다.

### 8.2 GitHub OAuth

- `GET /api/auth/login`에서 GitHub authorize URL로 리다이렉트한다.
- `GET /api/auth/callback`에서 `code`를 access token으로 교환한다.
- 서버가 GitHub 사용자 정보를 조회한다.
- `users` 테이블에 사용자 정보를 upsert한다.
- GitHub access token은 `user_oauth_tokens` private table에 저장한다.
- httpOnly 세션 쿠키를 발급한다.
- access token은 클라이언트, API 응답, page props, 로그에 노출하지 않는다.
- 로그아웃을 제공한다.

### 8.3 대시보드

- 로그인 사용자의 캐릭터를 보여준다.
- 현재 건강도(0~100)를 보여준다.
- 오늘 커밋 수를 보여준다.
- streak를 보여준다.
- 보유 포인트를 보여준다.
- 마지막 동기화 시각을 보여준다.
- 동기화가 throttle에 걸렸을 때 캐시 사용 상태를 이해할 수 있게 한다.
- 뽑기 가능 여부를 보여준다.
- 보유 아이템과 장착 아이템을 보여준다.
- 광고 슬롯 placeholder를 하단에 둔다.
- 방치 모드를 제공한다. 방치 모드는 방/캐릭터를 크게 보여주고 관리 UI를 최소화하는 보기다.
- 방치 모드에서는 참여 중인 룸이나 친구 상태를 작은 보조 신호로 보여줄 수 있다.
- MVP에서는 실제 광고 SDK를 연동하지 않고, 광고 영역은 향후 확장을 위해 구조적으로만 예약한다.
- 광고 placeholder는 필요하지 않으면 시각적으로 거의 드러내지 않는다.
- 방치 모드에서 광고 또는 페이지를 자동 새로고침하지 않는다.

### 8.4 커밋 동기화

- 대시보드 진입 시 서버에서 동기화를 시도한다.
- 마지막 동기화가 10분 이내면 기존 캐시를 사용한다.
- MVP는 `read:user` OAuth scope와 public PushEvent 집계를 사용한다.
- 비공개 커밋은 MVP에서 집계하지 않는다.
- GitHub API 오류가 발생하면 기존 캐시를 보여주고 오류를 사용자에게 과하게 노출하지 않는다.
- 신규 커밋에 대해서만 포인트를 지급한다.
- `commit_stats`로 날짜별 커밋 집계를 저장한다.

### 8.5 건강도

- 건강도는 0~100 정수다.
- 마지막 커밋 이후 경과 시간과 오늘 커밋 수, streak로 계산한다.
- 계산 결과는 `users.health`에 캐시한다.
- 건강도 구간에 따라 캐릭터 시각 상태가 바뀐다.

### 8.6 포인트

- 신규 커밋 1개당 10포인트를 지급한다.
- streak 유지 보너스로 하루 추가될 때 5포인트를 지급한다.
- 이미 집계된 날짜의 커밋에는 중복 지급하지 않는다.

### 8.7 뽑기

- 뽑기 1회 비용은 100포인트다.
- 등급별 확률은 common 70%, rare 25%, unique 5%다.
- 같은 등급 안에서는 균등 랜덤으로 아이템을 선택한다.
- 이미 보유한 아이템이 나오면 30포인트를 환급한다.
- 뽑기 결과는 인벤토리에 반영한다.
- 뽑기 연출은 MVP에서 간단한 카드 공개 방식으로 충분하다.

### 8.8 인벤토리와 장착

- 보유 아이템을 등급과 슬롯 정보와 함께 보여준다.
- 슬롯은 `hat`, `face`, `accessory`, `background`다.
- 한 슬롯에는 아이템 1개만 장착할 수 있다.
- 장착, 교체, 해제를 지원한다.
- 장착 결과는 캐릭터 렌더러에 반영한다.

### 8.9 룸

- 로그인 사용자는 룸을 만들 수 있다.
- 룸 생성 시 짧은 공유 코드가 발급된다.
- 생성자는 자동으로 첫 멤버가 된다.
- 사용자는 코드 입력 또는 `/room/[code]` 링크로 룸에 즉시 참여할 수 있다.
- 한 번 참여한 룸은 내 룸 목록에서 다시 입장할 수 있다.
- 룸 페이지는 멤버 캐릭터를 건강도 순으로 보여준다.
- 각 멤버 카드에는 GitHub login, 오늘 커밋 수, streak, 건강도를 보여준다.
- 사용자는 여러 룸에 참여할 수 있다.
- 룸 나가기를 제공한다.
- 룸 삭제는 MVP 제외다.
- 친구 추가/친구 요청 없이 룸 멤버십만으로 접근을 관리한다.
- 공동 룸에는 멤버끼리만 볼 수 있는 간단한 텍스트 채팅을 제공한다.
- MVP 채팅은 메시지 작성, 최근 메시지 조회, 작성자 login/시각 표시를 지원한다.
- MVP 채팅은 웹소켓 기반 고급 실시간 인프라를 필수로 하지 않는다. 구현은 짧은 폴링 또는 수동 갱신으로 시작할 수 있다.
- 채팅 메시지는 1~300자 텍스트로 제한한다.
- 디비의 말풍선이나 시스템 대사는 공동 룸 채팅에 끼어들지 않는다.
- 채팅 메시지 삭제, 수정, 신고, 차단, 멘션, 파일 첨부, 읽음 표시는 MVP 제외다.

## 9. 건강도와 포인트 계산

### 9.1 건강도 시작 로직

서버 동기화 시 아래 기본값으로 시작한다. 플레이 테스트 후 조정할 수 있지만, 변경 시 `docs/DECISIONS.md`에 기록한다.

```text
health = 50
health -= floor(hoursSinceLastCommit / 6) * 10
health += todayCommitCount * 15
health += min(currentStreakDays * 5, 25)
health = clamp(health, 0, 100)
```

### 9.2 건강도 구간

| 건강도 | 상태 이름 | 시각 표현 |
|---|---|---|
| 80~100 | 최상 | 밝은 표정, 깔끔한 얼굴, 반짝임 |
| 50~79 | 보통 | 평범한 표정 |
| 20~49 | 나쁨 | 피곤한 표정, 어두운 톤 |
| 0~19 | 방치 | 심하게 흐트러진 상태, 작은 방치 연출 |

### 9.3 포인트 지급

```text
pointsDelta = newCommitCount * 10 + newStreakDays * 5
```

중복 지급 방지 원칙:

- 날짜별 커밋 수는 `commit_stats`에 저장한다.
- 이미 저장된 날짜가 있으면 그 날짜의 기존 집계와 새 집계 차이만 보상한다.
- 커밋 삭제 또는 GitHub API 결과 감소로 포인트를 회수하지 않는다.

## 11. 화면 구성

| 라우트 | 이름 | 역할 |
|---|---|---|
| `/` | 랜딩 | 제품 소개, GitHub 로그인 진입 |
| `/dashboard` | 내 대시보드 | 캐릭터, 건강도, 포인트, 동기화, 뽑기, 인벤토리, 장착 |
| `/room/[code]` | 룸 | 룸 멤버 캐릭터 그리드, 공동 룸 채팅, 참여/나가기 |

광고 슬롯:

- `/dashboard` 하단에 placeholder를 둔다.
- `/room/[code]`에는 넓은 화면에서 사이드 placeholder를 둘 수 있다.
- 실제 광고 연동은 하지 않는다.

## 12. 비기능 요구사항

- 무료 티어에서 동작해야 한다.
- 로컬 개발은 `.env.local`만 있으면 가능해야 한다.
- 브라우저에 secret이 노출되면 안 된다.
- 모바일 화면에서 핵심 정보가 접히거나 겹치면 안 된다.
- GitHub API 실패 시 전체 앱이 죽으면 안 된다.
- 동기화와 보상 계산은 재시도해도 중복 포인트를 지급하면 안 된다.
- MVP 단계에서는 빠른 개발을 위해 테스트 범위를 핵심 순수 로직과 API 경계 중심으로 잡는다.

## 13. 열린 항목과 후속 확장

아래 항목은 MVP 진행을 막지 않는다. 확정되면 `docs/DECISIONS.md`에 기록하고 관련 문서를 업데이트한다.

- 후속 확장: 비공개 커밋 집계를 위해 `repo` scope 또는 GitHub App을 도입할지 여부
- 플레이 테스트 항목: 건강도 decay 수치가 너무 강한지 여부. MVP 기본값은 6시간마다 -10, 오늘 커밋당 +15, streak 최대 +25다.
- 플레이 테스트 항목: 중복 아이템 환급 30포인트가 적절한지 여부. MVP 기본값은 30포인트 환급이다.
