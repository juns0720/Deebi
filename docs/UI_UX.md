# UI_UX.md — Deebi UI/UX 설계

이 문서는 Deebi의 화면, 사용자 흐름, 시각 방향을 정의한다. 아직 확정되지 않은 취향과 브랜드 방향은 `docs/UI_UX_QUESTIONS.md`에서 개발자와 질의응답으로 확정한다.

---

## 1. UX 목표

- 사용자는 5초 안에 "내 커밋 활동이 캐릭터 상태로 바뀐다"는 컨셉을 이해해야 한다.
- 로그인 후 첫 화면에서 오늘 할 일을 바로 알 수 있어야 한다.
- 건강도, 오늘 커밋 수, streak, 포인트가 한눈에 보여야 한다.
- 뽑기와 장착은 작고 즐거운 보상이어야 하며, 업무 도구처럼 복잡하면 안 된다.
- 룸은 경쟁보다 가벼운 비교와 응원에 가까운 느낌이어야 한다.
- 모바일에서도 대시보드 핵심 정보가 위쪽에 먼저 보여야 한다.

## 2. UI 성격

확정된 방향:

- Deebi는 픽셀 캐릭터를 중심으로 한 가벼운 방치형 서비스다.
- 사용자는 캐릭터를 계속 조작하기보다, 일하거나 놀면서 같은 공간에 둔 캐릭터를 자연스럽게 본다.
- 친구들의 캐릭터와 내 캐릭터가 한 공간에 있는 느낌이 중요하다.
- 룸은 경쟁장보다 함께 켜두는 작업/놀이 공간에 가깝다.
- 전체 UI는 복잡한 게임 대시보드가 아니라, 픽셀 캐릭터가 살아 있는 간단한 공간이어야 한다.
- 전체 UI/UX는 픽셀 그래픽 감성으로 구현한다. 캐릭터만 픽셀인 것이 아니라 버튼, 패널, 탭, 배지, 채팅, 방 오브젝트까지 같은 픽셀 세계 안에 있어야 한다.
- 개발 활동 데이터는 캐릭터 상태를 움직이는 배경 규칙으로 존재하되, 화면의 감정적 중심은 캐릭터와 룸이다.

구현 중 조정 가능:

- 개인 방과 공동 룸의 세부 픽셀 오브젝트 배치는 Phase 02 prototype에서 실제 화면을 보며 조정한다.
- 단, 내 방/공동 룸/작업 테이블/문/8명 상한/채팅 배치 같은 구조적 결정은 확정된 기준으로 본다.

질문은 `docs/UI_UX_QUESTIONS.md`를 따른다.

## 2.1 공간 모델

확정된 방향:

- 기본 상태는 사용자의 **내 방**이다.
- 사용자는 자기 방에서 내 캐릭터의 건강도, 활동 상태, 아이템, 포인트를 본다.
- 친구와 함께할 때는 별도 **공동 룸**으로 들어간다.
- 공동 룸은 각자 캐릭터 카드만 나열하는 화면보다, 하나의 큰 테이블 또는 중심 오브젝트 주변에 캐릭터들이 모여 있는 느낌을 우선한다.
- 이 구조는 "혼자 켜두는 방치형 캐릭터"와 "친구들과 같이 개발하는 공간"을 분리해준다.

UX 의미:

- 내 방: 개인 루틴 확인, 캐릭터 관리, 뽑기와 장착
- 공동 룸: 친구들과 같은 공간에 접속해 일하고 노는 느낌, 서로의 상태를 가볍게 확인
- 공동 룸은 경쟁장보다 "같이 앉아 있는 개발 테이블"에 가깝게 설계한다.

내 방 방향:

- 내 방은 **꾸미기 중심 개인 방**으로 설계한다.
- MVP에서는 방 자체를 꾸미는 기능을 구현하지 않지만, 배경/가구/소품 확장이 자연스럽게 들어갈 수 있는 구도를 잡는다.
- 캐릭터는 방의 중심에 있고, 장착 아이템과 상태 변화가 먼저 보인다.
- 방은 단순한 장식 배경이 아니라 "내 캐릭터가 머무는 곳"으로 느껴져야 한다.
- 추후 배경 아이템(`background` slot)이나 가구형 아이템이 들어와도 구조를 크게 바꾸지 않도록 한다.
- 방 꾸미기 확장은 **배치형 가구 아이템**까지 고려한다. 예: 책상, 의자, 식물, 포스터, 조명, 작은 개발 장비.
- 단, MVP에서는 사용자가 직접 좌표를 배치하는 격자 편집기는 만들지 않는다.
- 초기 구조는 정해진 앵커 위치에 가구/소품 레이어를 꽂을 수 있는 방식으로 열어둔다.

시각 후보:

- 내 방은 벽, 바닥, 작은 작업 책상, 선반, 창문 같은 픽셀 룸 요소를 얕게 둔다.
- MVP의 `background` 슬롯은 방 분위기를 바꾸는 첫 확장 지점으로 취급한다.
- 향후 가구 아이템은 `room_object` 또는 유사 슬롯으로 확장할 수 있게 명명과 렌더링 구조를 막지 않는다.
- 중앙에 큰 작업 테이블, 모니터, 노트북, 머그컵 같은 픽셀 오브젝트를 둔다.
- 캐릭터들은 테이블 주변 좌석 또는 바닥 타일 위에 배치한다.
- 멤버 수가 늘어나면 원형/반원형 배치에서 그리드 배치로 자연스럽게 전환한다.
- 건강도 순위는 강한 랭킹 UI보다 배치 순서, 작은 상태 배지, 캐릭터 표정으로 은근히 드러낸다.

공동 룸 배치:

- 공동 룸은 멤버 수에 따라 자동으로 배치를 바꾼다.
- 공동 룸의 중심 오브젝트는 **큰 작업 테이블**이다.
- 적은 인원은 큰 작업 테이블 주변에 모여 있는 느낌을 우선한다.
- 인원이 늘어나면 각자 작은 책상 또는 작업 자리처럼 보이는 배치로 전환한다.
- 너무 많은 캐릭터가 한 화면에 들어와 공간감과 캐릭터 식별성이 무너지지 않도록 MVP에서 룸 멤버 수 상한을 둔다.
- MVP 룸 멤버 수 상한은 **8명**이다.
- 권장 배치 단계:
  - 1~4명: 중앙 테이블 주변 배치
  - 5~8명: 양쪽 좌석, 반원형 작업 테이블, 또는 작은 책상 배치
  - 9명 이상: MVP에서는 참여 제한

공동 룸 테이블 기준:

- 큰 작업 테이블은 공동 룸의 시각적 중심이다.
- 테이블 위에는 노트북, 키보드, 머그컵, 작은 문서, 간단한 조명 같은 픽셀 오브젝트를 둘 수 있다.
- 캐릭터는 테이블 주변 좌석 또는 가까운 작업 위치에 배치한다.
- 테이블은 멤버 수가 늘어도 공간의 중심을 유지해야 한다.
- 5~8명에서는 테이블이 길어지거나 양쪽 좌석이 늘어나는 방식으로 보이게 한다.
- 친구 활동 상태는 캐릭터 라벨과 테이블 오브젝트/이펙트 조합으로 표현한다.

## 3. 정보 구조

### 3.1 로그인 전

| 영역 | 목적 |
|---|---|
| 브랜드 | Deebi 이름과 픽셀 심볼 인지 |
| 컨셉 문장 | 커밋 -> 캐릭터 건강도 연결 설명 |
| GitHub 로그인 | 유일한 주요 CTA |
| 미리보기 | 건강한 캐릭터와 핵심 지표 예시 |
| 준비/신뢰 정보 | GitHub OAuth, 비공개 정보 보호 설명 |

### 3.2 로그인 후 대시보드

대시보드는 상단 탭으로 `내 방`과 `함께하는 방`을 분리한다.

| 탭 | 역할 |
|---|---|
| `내 방` | 내 캐릭터, 건강도, 커밋 지표, 포인트, 뽑기, 인벤토리, 장착 |
| `함께하는 방` | 참여 중인 룸 목록, 방 만들기, 코드로 참여, 최근 입장 |

`내 방` 우선순위:

| 우선순위 | 정보 |
|---|---|
| 1 | 캐릭터 현재 상태와 개인 방 |
| 2 | 건강도, 오늘 커밋, streak, 포인트 |
| 3 | 동기화 상태와 마지막 동기화 시각 |
| 4 | 뽑기 가능 여부 |
| 5 | 장착 중 아이템 |
| 6 | 인벤토리 |
| 7 | 광고 placeholder |

`내 방` 데스크톱 레이아웃:

- 데스크톱은 방치형으로 켜두는 사용성을 우선한다.
- 화면의 중심은 큰 **방 스테이지**다.
- 방 스테이지는 전체 콘텐츠 폭의 약 65~70%를 차지한다.
- 오른쪽에는 320~380px 폭의 관리 패널을 둔다.
- 관리 패널에는 건강도, 오늘 커밋, streak, 포인트, 동기화 상태, 뽑기 CTA, 장착 슬롯을 배치한다.
- 인벤토리는 하단 dock 또는 접이식/스크롤 영역으로 둔다.
- 방 스테이지는 최소 560px 이상 확보하는 것을 목표로 한다.
- 추후 집중/방치 모드를 만들 경우 오른쪽 패널을 접고 방만 크게 보는 구조로 확장할 수 있게 한다.

`내 방` 모바일 레이아웃:

- 위에서부터 방/캐릭터, 핵심 상태 카드, 뽑기 CTA, 장착 슬롯, 인벤토리 순서로 쌓는다.
- 모바일에서는 오른쪽 패널을 별도로 만들지 않고 세로 흐름으로 변환한다.
- 인벤토리는 접이식 또는 탭 내부 영역으로 길이를 제어한다.

방치 모드:

- MVP에 간단한 **방치 모드**를 포함한다.
- 방치 모드는 사용자가 Deebi를 켜둔 채 일하거나 놀 때 방/캐릭터를 크게 보는 화면이다.
- 방치 모드에서는 관리 패널, 인벤토리, 보조 CTA를 숨기거나 최소화한다.
- 핵심으로 남기는 정보는 캐릭터, 방, 건강도 요약, 오늘 커밋 수, streak, 현재 참여 중인 룸 상태 정도다.
- 방치 모드는 **내 방 + 친구 상태 미니 신호** 형태로 설계한다.
- 기본 화면은 내 방과 내 캐릭터를 크게 보여준다.
- 사용자가 참여 중인 룸이나 친구들이 있으면, 화면 한쪽에 작은 상태 알림 또는 미니 표시를 둘 수 있다.
- 친구 관련 표시는 방치 모드의 중심이 아니라 보조 신호다.
- 친구 상태 미니 신호는 일반 알림 UI보다 **방 안 오브젝트**로 표현하는 방향을 우선한다.
- 예: 문, 창문, 작은 전광판, 데스크 램프, 벽걸이 모니터 같은 픽셀 오브젝트.
- 오브젝트는 `함께하는 방 2개`, `지금 3명이 작업 중`, `민수 건강도 회복` 같은 정보를 은근하게 표시할 수 있다.
- 텍스트를 길게 띄우기보다 아이콘, 숫자, 짧은 라벨, 작은 점등 효과 중심으로 표현한다.
- 접근성을 위해 오브젝트 hover/focus 또는 별도 텍스트 영역에서 의미를 확인할 수 있어야 한다.

공동 룸 진입 오브젝트:

- 내 방에서 공동 룸으로 이동하는 대표 오브젝트는 **문**이다.
- 문은 `함께하는 방` 탭 또는 최근 공동 룸으로 이어지는 공간적 CTA 역할을 한다.
- 참여 중인 룸이 있으면 문에 작은 불빛, 숫자 배지, 손잡이 반짝임 등으로 상태를 표시할 수 있다.
- 참여 중인 룸이 없으면 문은 방 만들기/코드 참여로 이어지는 진입점이 될 수 있다.
- 문은 게임적 포털처럼 과하게 보이기보다, 내 방 밖에 친구 작업 공간이 있다는 느낌을 준다.
- 키보드 접근과 명확한 라벨을 제공한다. 예: `함께하는 방으로 이동`.

## 2.2 색상 분위기

확정된 기본 방향:

- Deebi의 기본 색상 분위기는 **밝은 픽셀 방**이다.
- 기본 배경은 따뜻한 크림/아이보리 계열을 사용한다.
- 주요 포인트는 초록/파랑 계열로 시작한다.
- 캐릭터와 아이템이 화면의 주인공이 되도록 UI 색은 과하게 튀지 않는다.
- 방 배경은 추후 커스텀 가능하므로, 기본 팔레트는 확장 가능한 중립적 따뜻함을 유지한다.

확장 방향:

- `background` 아이템으로 밤, 도시, 우주, 창문, 계절감 같은 분위기를 바꿀 수 있다.
- 기본 UI 토큰은 배경 아이템이 바뀌어도 읽기 쉬워야 한다.
- 배경 아이템이 강한 색을 써도 정보 패널과 라벨은 충분한 대비를 유지한다.

초기 팔레트 후보:

| 역할 | 색감 |
|---|---|
| 앱 배경 | warm ivory, soft cream |
| 방 바닥/벽 | pale wood, light beige, muted mint |
| 주요 포인트 | green, blue |
| 경고/방치 | muted amber, dusty rose, soft gray |
| 텍스트 | near black, slate |
| 카드/패널 | white 또는 아주 옅은 warm gray |

## 2.3 픽셀 그래픽과 타이포그래피

확정된 방향:

- Deebi의 모든 주요 UI는 **픽셀 그래픽 감성**으로 구현한다.
- 앱 기본 폰트는 **PF스타더스트**를 사용한다.
- PF스타더스트는 랜딩, 내 방, 공동 룸, 채팅, 버튼, 탭, 라벨, 숫자 표시까지 기본 폰트로 적용한다.
- 폰트 파일은 가능하면 프로젝트에 로컬 파일로 포함하고 `next/font/local` 또는 동등한 방식으로 로드한다.
- PF스타더스트 파일은 `public/fonts/PFStardust.woff`에 포함되어 있다.
- Phase 02에서는 이미 포함된 font-face를 기준으로 디자인 토큰, fallback, 크기/줄간격을 정리한다.
- 배포 전에는 라이선스와 웹 임베딩 가능 여부를 다시 확인한다.

픽셀 UI 기준:

- 버튼, 패널, 탭, 배지, 채팅 말풍선은 부드러운 SaaS 카드보다 픽셀 게임 UI에 가까운 형태를 우선한다.
- 모서리는 과하게 둥글게 만들지 않는다. 픽셀 감성을 살리는 직각 또는 작은 radius를 사용한다.
- 그림자는 blur가 큰 현대식 그림자보다 1~3px 단단한 offset shadow, pixel border, inset line을 우선한다.
- 아이콘은 가능한 한 픽셀 스타일 또는 픽셀 감성에 어울리는 단순한 형태를 사용한다.
- 배경, 버튼, 카드에 과한 gradient, bokeh, glassmorphism, 부드러운 3D 효과를 쓰지 않는다.
- 공간감은 픽셀 방의 벽/바닥/가구/조명/오브젝트로 만든다.
- UI 요소는 픽셀 감성을 가지되, 텍스트 가독성과 클릭 영역은 일반 웹 접근성 기준을 유지한다.
- 작은 라벨과 긴 GitHub login은 PF스타더스트에서도 읽히도록 충분한 크기, 줄간격, 대비를 확보한다.
- 데스크톱에서는 `방치 모드` 버튼으로 오른쪽 관리 패널을 접고 방 스테이지를 넓힌다.
- 모바일에서는 전체 화면에 가까운 방/캐릭터 보기로 전환하되, 종료 버튼은 항상 보여야 한다.
- 방치 모드는 향후 광고 수익 가능성을 고려하되, MVP에서는 광고 위치를 적극 노출하지 않는다.
- 광고 영역은 화면 구조상 들어갈 수 있는 안전한 위치만 예약한다.
- MVP에서는 실제 광고 SDK를 연동하지 않고 placeholder만 둔다.
- 필요하지 않으면 placeholder도 시각적으로 거의 드러내지 않는다.
- 방치 모드가 광고 노출을 인위적으로 늘리는 장치처럼 동작하면 안 된다.
- 페이지나 광고 영역을 사용자 요청 없이 자동 새로고침하지 않는다.
- 광고 클릭을 유도하는 카피, 보상, UI 배치는 금지한다.

### 3.3 룸

| 우선순위 | 정보 |
|---|---|
| 1 | 룸 이름과 공유 코드 |
| 2 | 내 캐릭터 또는 내 순위 |
| 3 | 멤버 캐릭터 그리드 |
| 4 | 각 멤버의 login, 건강도, 오늘 커밋, streak |
| 5 | 공동 룸 채팅 |
| 6 | 룸 나가기 |
| 7 | 광고 placeholder |

룸 멤버 상태 표현:

- 공동 룸에서는 라벨과 오브젝트/이펙트를 함께 사용한다.
- 기본 정보는 캐릭터 근처 작은 라벨로 표시한다. 예: GitHub login, 오늘 커밋 수, streak, 건강도 구간.
- 분위기 표현은 테이블 위 노트북 불빛, 머그컵, 작은 sparkle, 피곤한 상태 효과 같은 픽셀 오브젝트/이펙트로 보조한다.
- 라벨은 항상 읽을 수 있어야 하며, 오브젝트/이펙트만으로 중요한 정보를 전달하지 않는다.
- 룸 전체가 숫자 대시보드처럼 보이지 않도록 라벨은 작고 정돈된 밀도로 유지한다.

공동 룸 채팅:

- 공동 룸에는 룸 멤버끼리만 볼 수 있는 간단한 텍스트 채팅을 둔다.
- 채팅은 친구들과 같은 방에 있는 느낌을 보강하는 보조 영역이다. 화면의 중심은 여전히 픽셀 룸, 큰 작업 테이블, 캐릭터다.
- 데스크톱에서는 룸 스테이지 오른쪽에 320~360px 고정 폭 패널로 배치한다.
- 모바일에서는 하단 시트 형태로 연다.
- 데스크톱 오른쪽 패널은 룸 스테이지를 밀어내되, 캐릭터와 큰 작업 테이블이 화면 중심에서 사라지면 안 된다.
- 모바일 하단 시트는 닫힘/열림 상태를 제공하고, 닫힌 상태에서도 새 메시지 여부를 작은 배지나 라벨로 알 수 있게 한다.
- 하단 시트가 열려도 메시지 입력창과 닫기 버튼은 안전 영역 안에 있어야 한다.
- 메시지 목록은 최근 메시지 50개를 기본으로 보여준다.
- 메시지 입력창은 한 줄 입력으로 시작하되, 긴 문장은 2~3줄까지 자연스럽게 늘어날 수 있다.
- 메시지는 작성자 GitHub login, 짧은 시각, 본문을 보여준다.
- 내 메시지와 친구 메시지는 위치나 배경 톤으로 구분하되, 과한 메신저 앱처럼 보이지 않게 한다.
- 디비의 말풍선이나 시스템성 대사는 공동 룸 채팅에 끼어들지 않는다.
- 빈 상태 카피는 친구에게 말을 걸 수 있게 가볍게 쓴다. 예: `아직 조용해요. 먼저 한마디 남겨볼까요?`
- 입력 placeholder 예시: `친구들에게 짧게 말하기`
- 전송 버튼은 `보내기` 또는 send 아이콘으로 표시한다.
- 신고/차단/멘션/파일 첨부/읽음 표시는 MVP UI에 노출하지 않는다.

## 4. 화면별 설계

### 4.1 `/` 랜딩

필수 요소:

- Deebi 로고 텍스트
- 픽셀 심볼 또는 캐릭터 미리보기
- 핵심 문장
- GitHub 로그인 버튼
- OAuth 구현 전에는 준비 중 또는 비활성 상태로 표시 가능
- 서비스가 GitHub activity를 사용한다는 설명
- 친구들과 같은 방에서 개발하는 공동 룸 장면 미리보기

금지:

- 실제 구현되지 않은 기능을 가능한 것처럼 CTA로 노출하지 않는다.
- 광고, 결제, 상점 같은 MVP 제외 기능을 강조하지 않는다.

랜딩 톤:

- 랜딩은 짧은 소개형 + 공동 룸 미리보기 조합으로 설계한다.
- 첫 화면에서 친구들과 같은 방에서 개발하는 픽셀 공간을 보여준다.
- 제품 설명은 길게 늘이지 않고, 핵심 루프를 짧게 전달한다.
- 주요 메시지는 "각자 코딩해도, 같은 방에 있는 것처럼."이다.
- GitHub 로그인은 명확한 1차 CTA로 둔다.
- 랜딩은 마케팅 페이지보다 앱 경험의 미리보기에 가깝게 만든다.

랜딩 첫 화면 구성 후보:

- 상단: Deebi 로고, GitHub 로그인
- 중앙: 공동 작업 테이블 룸 미리보기
- H1: `각자 코딩해도, 같은 방에 있는 것처럼.`
- 보조 문장: `Deebi는 GitHub 커밋으로 캐릭터 상태가 바뀌는 작은 픽셀방이에요. 내 방을 켜두고, 친구들과 같은 방에서 느슨하게 같이 개발해요.`
- 하단 힌트: 내 방, 뽑기, 함께하는 방의 짧은 3단계 루프

- 모바일에서는 탭을 화면 상단에 고정하지 않아도 되지만, 첫 화면에서 쉽게 전환 가능해야 한다.
- 데스크톱에서는 탭 아래 콘텐츠가 넓게 펼쳐지고, 사이드바는 필수로 두지 않는다.

### 4.3 `/room/[code]`

필수 섹션:

- 룸 헤더
- 공유 코드 복사
- 멤버 수
- 멤버 캐릭터 그리드
- 공동 룸 채팅 패널
- 로그인 사용자가 코드/링크로 접근하면 즉시 참여
- 참여 중인 사용자의 나가기 버튼
- 룸을 찾을 수 없음 상태
- 내가 참여한 룸은 대시보드의 내 룸 목록에서 다시 입장 가능

정렬:

- 건강도 높은 순
- 오늘 커밋 수 높은 순
- login 가나다/알파벳 순

참여 UX:

- MVP에서는 코드 또는 링크를 아는 로그인 사용자가 바로 룸에 참여한다.
- 별도 친구 추가, 친구 요청, 승인 플로우는 만들지 않는다.
- 한 번 참여한 룸은 `room_members`에 남으므로, 사용자는 대시보드의 내 룸 목록에서 언제든 다시 입장할 수 있다.
- 사용자가 룸을 나가면 내 룸 목록에서도 사라진다.
- 이 방식은 "친구 방에 자유롭게 놀러가기" 느낌을 주되, MVP에서 소셜 그래프를 만들지 않는 원칙을 지킨다.

## 5.1 픽셀 캐릭터 렌더링 기준

확정된 방향:

- 캐릭터 기본 해상도는 **32x32**다.
- 캐릭터의 최종 디자인은 개발자가 별도로 제작해 제공한다.
- 에이전트는 최종 캐릭터 생김새, 종족, 얼굴형, 실루엣을 임의로 확정하지 않는다.
- 캐릭터 디자인이 제공되기 전에는 placeholder 또는 구조 검증용 임시 픽셀 캐릭터만 사용한다.
- 32x32를 기준으로 표정, 수염/방치 상태, 얼굴 아이템, 모자, 액세서리를 표현한다.
- 렌더링은 원본 픽셀 해상도를 유지하고 CSS로 정수 배율 확대한다.
- 브라우저에서는 `image-rendering: pixelated` 또는 SVG grid 방식으로 픽셀 선명도를 유지한다.
- 모바일에서도 최소 160px 이상으로 보여 캐릭터 상태를 알아볼 수 있게 한다.
- 대시보드의 대표 캐릭터는 192~256px 범위에서 반응형으로 조절한다.
- 룸 멤버 캐릭터는 멤버 수에 따라 96~160px 범위로 줄일 수 있다.

레이어 기준:

- `background`: 방/배경 분위기
- `base`: 캐릭터 몸과 얼굴 기본형
- `health_state`: 건강도별 표정, 수염, 방치 연출
- `face`: 안경, 선글라스, 안대, 외알안경
- `hat`: 모자, 비니, 머리띠, 왕관
- `accessory`: 커피잔, 헤드폰, 메달 등 주변 소품
- `effect`: 최상 상태 반짝임, 방치 상태 작은 효과

아이템 애니메이션 기준:

- 아이템은 종류에 따라 캐릭터 애니메이션을 따라갈지 결정한다.
- `hat`, `face`처럼 캐릭터 몸에 직접 붙는 아이템은 캐릭터 idle 움직임을 따라간다.
- 손에 들거나 몸 가까이에 있는 일부 `accessory`는 캐릭터 움직임을 따라갈 수 있다.
- `background`는 캐릭터 움직임과 독립적으로 고정된다.
- 향후 가구/방 소품 아이템은 방 좌표계에 고정된다.
- `effect`는 캐릭터 상태와 연결되지만, 캐릭터에 붙는 효과와 방에 떠 있는 효과를 구분할 수 있어야 한다.
- MVP에서 애매한 accessory는 우선 고정 또는 약한 동기화 중 하나를 item metadata로 명시한다.

주의:

- 32x32 내부 좌표계는 Phase 06에서 코드 상수로 고정한다.
- Phase 06의 렌더러는 개발자가 제공한 캐릭터 시안을 반영할 수 있도록 레이어 입력과 좌표계를 분리한다.
- 아이템은 캐릭터 좌표계를 기준으로 제작하되, 방/가구 아이템은 별도 앵커 좌표계를 사용할 수 있게 열어둔다.
- 너무 작은 디테일은 모바일에서 뭉개지므로 1픽셀 장식에 의존하지 않는다.

애니메이션 요구:

- 캐릭터는 상태별로 구분되어야 한다.
- 캐릭터는 완전히 정지된 이미지가 아니라, 간단한 반복 움직임을 가질 수 있어야 한다.
- MVP에서 필요한 최소 애니메이션은 idle 계열이다. 예: 숨쉬기, 눈 깜빡임, 살짝 흔들림, 방치 상태의 작은 상태 효과.
- 애니메이션은 사용자가 직접 조작하는 액션 게임처럼 빠르거나 복잡하지 않아야 한다.
- 상태별 애니메이션은 health 구간과 연결한다.
- 애니메이션 강도는 **살짝 살아 있는 정도**로 제한한다.
- 기본 캐릭터는 몸이 살짝 흔들리거나 1~2픽셀 호흡하고, 가끔 표정/눈 깜빡임이 들어가는 정도가 적절하다.
- 건강한 상태라고 크게 튀거나, 방치 상태라고 과하게 움직이는 연출은 피한다.
- 오래 켜두어도 시야를 피곤하게 만들지 않는 속도와 반복감을 우선한다.
- 상태별 캐릭터 애니메이션의 구체적 디자인은 개발자가 별도로 제작한다.
- 에이전트는 건강/보통/피곤/방치 상태의 최종 포즈, 표정, 실루엣을 임의로 디자인하지 않는다.
- 구현은 health 구간과 개발자가 제공한 애니메이션 state를 정확히 연결하는 데 집중한다.

권장 에셋 방식:

- 1차 권장안은 **상태별 스프라이트시트 PNG + manifest JSON**이다.
- 예: `healthy_idle`, `normal_idle`, `tired_idle`, `neglected_idle` 상태를 각각 2~4프레임으로 둔다.
- manifest는 frame width, frame height, frame count, fps, state name을 정의한다.
- 기본 fps는 3~5 범위로 시작한다.
- 개발자가 PNG 스프라이트시트를 제공하면 에이전트는 이를 렌더러에 연결한다.
- 개발자가 개별 PNG 여러 장을 제공해도 빌드 단계에서 스프라이트시트처럼 다룰 수 있게 구조를 열어둔다.
- SVG/좌표 데이터 방식은 색상 변형은 쉽지만, 개발자가 직접 캐릭터를 제작하는 현재 흐름에서는 우선순위를 낮춘다.

health state 매핑:

| health | state key | 의미 |
|---|---|---|
| 80~100 | `healthy_idle` | 최상 상태 |
| 50~79 | `normal_idle` | 보통 상태 |
| 20~49 | `tired_idle` | 나쁨/피곤 상태 |
| 0~19 | `neglected_idle` | 방치 상태 |

## 6. 반응형 원칙

모바일:

- 1열 레이아웃
- 캐릭터와 핵심 지표를 가장 먼저 보여준다.
- 인벤토리는 탭 또는 접이식으로 내려도 된다.
- 룸 멤버 카드는 1~2열로 조정한다.

데스크톱:

- 대시보드 `내 방`은 큰 방 스테이지와 오른쪽 관리 패널을 기본으로 한다.
- 룸은 멤버 그리드를 넓게 보여준다.
- 광고 placeholder는 콘텐츠를 방해하지 않는 위치에 둔다.

공통:

- 텍스트가 카드나 버튼 밖으로 넘치면 안 된다.
- 버튼 높이와 카드 크기는 상태 변화로 흔들리지 않게 한다.
- 숫자와 status badge는 긴 텍스트에도 줄바꿈이 자연스러워야 한다.

## 7. 접근성 원칙

- 모든 버튼은 실제 `button` 또는 링크 의미에 맞는 요소를 쓴다.
- 색만으로 상태를 전달하지 않는다.
- 건강도 구간은 텍스트와 수치도 함께 제공한다.
- 뽑기 결과는 애니메이션 없이도 텍스트로 이해 가능해야 한다.
- 키보드 포커스가 보여야 한다.
- 모달은 escape 또는 닫기 버튼을 제공한다.

## 8. 애니메이션 원칙

허용:

- 캐릭터 idle 애니메이션
- 건강도 변화 시 짧은 반응
- 뽑기 카드 공개
- 작은 sparkle/상태 효과
- 방치 모드에서 방/캐릭터가 아주 미세하게 살아 있는 느낌

주의:

- 애니메이션은 정보 전달을 방해하지 않는다.
- 모바일에서 과도한 repaint를 만들지 않는다.
- 사용자가 reduced motion을 선호하면 핵심 애니메이션을 줄인다.
- 방치 모드 애니메이션은 오래 켜두어도 CPU/GPU 부담이 낮아야 한다.

## 9. UX 카피 원칙

- 한국어로 작성한다.
- 버튼은 짧고 행동 중심으로 쓴다.
- 오류는 사용자가 다음 행동을 알 수 있게 쓴다.
- GitHub 권한 설명은 솔직하게 쓴다.
- "곧 제공" 상태는 실제 다음 phase에서 구현될 기능에만 사용한다.
- Deebi의 기본 말투는 **살짝 친근하게** 간다.
- 기본 UI 숫자와 라벨은 읽기 쉽게 유지하되, 상태 메시지와 개인 방 말풍선은 부드럽게 쓴다.
- 너무 친구처럼 과하게 반말하거나 밈처럼 보이는 말투는 피한다.
- 디비가 직접 말하는 느낌의 카피는 개인 방에서만 사용한다.
- 공동 룸에서는 디비가 끼어들어 말하지 않고, 친구들끼리 대화하는 공간을 우선한다.

예시:

- 로그인: `GitHub로 시작하기`
- 동기화 중: `커밋 활동을 확인하는 중`
- 캐시 사용: `방금 확인해서 기존 결과를 보여줘요`
- 포인트 부족: `뽑기까지 {pointsNeeded}포인트 남았어요`
- 중복 환급: `이미 가진 아이템이라 30포인트를 돌려받았어요`
- 개인 방 상태: `오늘 3커밋 했어요`
- 개인 방 상태: `캐릭터가 꽤 멀쩡해요`

## 9.1 디비 말풍선과 공동 룸 채팅 경계

확정된 방향:

- 디비의 말풍선 또는 시스템성 대사는 **내 방**에서만 사용한다.
- 내 방에서 디비는 사용자의 커밋 상태, 건강도, 포인트, 뽑기 가능 여부를 살짝 친근하게 알려준다.
- 공동 룸에서는 디비가 사용자의 대화를 대신하거나 끼어들지 않는다.
- 공동 룸에는 친구들끼리 채팅할 수 있는 간단한 텍스트 채팅을 둔다.
- 채팅은 친구들이 같은 방에서 같이 개발하는 감각을 강화하는 기능으로 본다.

MVP 경계:

- 공동 룸 텍스트 채팅은 MVP에 포함한다.
- MVP 채팅은 룸 멤버 전용 메시지 작성/최근 메시지 조회 중심이다.
- 웹소켓 기반 고급 실시간 채팅은 필수가 아니며, 짧은 폴링 또는 수동 갱신으로 시작할 수 있다.
- DM, 신고/차단, 멘션, 파일 첨부, 읽음 표시는 MVP에서 만들지 않는다.

## 10. UI/UX 확정 전 금지사항

- 브랜드 로고를 최종안처럼 확정하지 않는다.
- 색상 팔레트를 최종 브랜드 색으로 고정하지 않는다.
- 캐릭터 해상도를 코드 전체에 되돌리기 어렵게 박지 않는다.
- 룸 경쟁 표현을 과하게 만들지 않는다.
- 사용자가 확정하지 않은 mascot 성격을 카피에 과도하게 반영하지 않는다.
